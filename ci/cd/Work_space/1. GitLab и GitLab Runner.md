# Поднимаем GitLab Runner

Чтобы поднять GitLab Runner, нам нужна обычная виртуальная машина, на которую мы его установим. Для подключения к ней нам понадобятся SSH-ключи. Если они у вас уже есть, вы можете использовать их. Если нет, то используйте этот скрипт. Вставьте код в свой терминал:

```bash
# Если директории ~/.ssh нет, то создаём с правами rwx только на овнера
#
if [ ! -d ~/.ssh ]; then mkdir ~/.ssh -m 0700; fi

# Проверяем, есть ли файлы id_ecdsa или id_ecdsa.pub Если они существуют, скрипт предложит вам 
# проверить директорию ~/.ssh на наличие ключей. Если же файлов нет, скрипт создаст пару ключей
# ecdsa: публичный и приватный. Размер ключей — 512 бит. Они будут без пароля и 
# с комментарием gitlab-cicd-course. Файлы сохранятся по пути ~/.ssh/id_ecdsa и ~/.ssh/id_ecdsa.pub
#
if [ -f ~/.ssh/id_ecdsa ] || [ -f ~/.ssh/id_ecdsa.pub ]; then 
printf "\033[31mПроверьте директорию ~/.ssh, возможно ключи уже есть.\033[m\n"
else
ssh-keygen -t ecdsa -b 521 -N "" -C gitlabcicdcourse -f ~/.ssh/id_ecdsa
fi
```

После его выполнения вы сможете скопировать содержимое публичного SSH-ключа, который будет находиться по пути `~/.ssh/id_ecdsa.pub`.

# Регистрируем GitLab Runner в GitLab

Вернёмся к ранее созданному GitLab. Мы добавим Instance Runner, который будет доступен для всех проектов.

Перейдём в админ панель:

![](https://ucarecdn.stepik.net/71796446-dd55-45b0-a62d-93f4abd6f9d2/)

Слева выберем «**CI/CD**», затем в открывшемся списке перейдём в раздел «**Runners**»:

![](https://ucarecdn.stepik.net/cf416f24-efee-4807-a031-f0d3740af271/)

Нажмём кнопку «**Сreate** **instance runne**r**»**:

![](https://ucarecdn.stepik.net/9920862a-862d-4cc1-b01d-6f8dd9bd95f0/)

Активируем опцию «**Run untagged jobs»**, чтобы запускать задачи без указания тегов (что это такое и как работает, мы разберём далее в курсе):

![](https://ucarecdn.stepik.net/24ef5a17-67cc-4288-bfdd-7c64f2bcf8c8/)

После чего нажмем кнопку «**Create runner****»**.

Отлично, мы создали раннер!

![](https://ucarecdn.stepik.net/6672e005-e6fd-424b-b0ef-8cd2d0c798ce/)

Следующий важный этап — установка самого GitLab Runner на виртуальную машину и его регистрация в GitLab.

Пролистаем ниже до «**Step 1»** и скопируем предлагаемую команду регистрации раннера:

![](https://ucarecdn.stepik.net/49d6ca32-e52f-46e3-a395-a26624ab8dc5/)

Далее нам потребуется ранее созданная виртуальная машина. Чтобы подключиться к ней, нам нужен её публичный IP-адрес, его можно найти в **`Консоли управления > Compute Cloud`** и прокрутить немного вправо y нужной машины:

![](https://ucarecdn.stepik.net/25e1b6ae-438b-404d-8bbc-db042e62ddf3/)

> **Примечание**: когда вы снова запустите машину после того, как остановили, её публичный IP-адрес может измениться. Если вы не можете подключиться, проверьте его.

Подключимся к ней по SSH (имя пользователя используйте то, которое вы указывали в поле «**Логин**» при создании машины), и выполним несколько команд по установке GitLab Runner:

```bash
# Добавляем официальный репозиторий GitLab
curl -L "https://packages.gitlab.com/install/repositories/runner/gitlab-runner/script.deb.sh" | sudo bash

# Обновляем списки пакетов
sudo apt-get update

# Устанавливаем раннер
sudo apt-get install gitlab-runner
```

Вставим ранее скопированную команду в терминал и запустим с правами суперпользователя (у вас будет свой URL и токен для регистрации):

```bash
ilyalushkevich@gitlab-runner:~$ sudo gitlab-runner register --url https://gitlabcicdcourse.gitlab.yandexcloud.net --token glrt-I1X7c-dEyYLpUBlc_lCOzG86MQp0OjEKdToyCw.01.121tpuu01
```

> **Примечание**: обязательно регистрируйте GitLab Runner с командой `sudo`. 
> 
> Когда вы регистрируете GitLab Runner через `sudo`, его параметры записываются в системный конфигурационный файл `/etc/gitlab-runner/config.toml`, который затем использует systemd-служба `gitlab-runner.service`.
> 
> Если же регистрация выполняется от имени обычного пользователя, параметры сохраняются в пользовательском файле `~/.gitlab-runner/config.toml`. В этом случае GitLab Runner не будет управляться системной службой, и его придется запускать вручную командой `gitlab-runner run`.

У нас запрашивают URL GitLab-инстанcа, поскольку он уже указан через флаг `--url`, жмём «Enter»:

```no-highlight
Runtime platform                                    arch=amd64 os=linux pid=3009 revision=9ffb4aa0 version=18.8.0
Running in system-mode.                            
                                                   
Enter the GitLab instance URL (for example, https://gitlab.com/):
[https://gitlabcicdcourse.gitlab.yandexcloud.net]:
```

На следующем шаге предлагается ввести имя для раннера, которое сохраняется в локальном файле `config.toml`. Вы можете указать уникальное имя или оставить поле пустым (тогда будет то, что в скобках), нажмём «Enter»:

```no-highlight
Verifying runner... is valid                        correlation_id=01KHVMD3PA19N5651P10NWBR4V runner=I1X7c-dEy
Enter a name for the runner. This is stored only in the local config.toml file:
[gitlab-runner]:
```

Теперь мы должны указать executor, который будет использоваться для выполнения задач. Укажем `docker`, т. к. мы будем использовать его:

```bash
Enter an executor: kubernetes, instance, custom, ssh, docker+machine, docker-autoscaler, shell, parallels, virtualbox, docker, docker-windows:
docker # Указываем docker
```

Заключительным шагом нам предлагают указать Docker-образ по умолчанию, который будет использоваться, если мы явно не будем его указывать. Давайте выберем `ubuntu`:

```bash
Enter the default Docker image (for example, ruby:3.3):
ubuntu # Указываем нужный Docker-образ
```

Мы успешно зарегистрировали раннер!

```no-highlight
Runner registered successfully. Feel free to start it, but if it's running already the config should be automatically reloaded!
 
Configuration (with the authentication token) was saved in "/etc/gitlab-runner/config.toml"
```

![](https://ucarecdn.stepik.net/c196ccec-45bd-4948-9d37-f85908f25d78/)
