**GitLab Runner** — это приложение, написанное на Go, которое выполняет задачи, описанные в файле `.gitlab-ci.yml`. Он забирает их из GitLab через API и выполняет в указанной среде: сборка, тестирование, деплой — всё, что вы укажете. Без Runner ваша конфигурация не сможет запускаться, так как именно он превращает инструкции в реальные действия.
# Как работает GitLab Runner?

GitLab Runner действует как агент, который постоянно проверяет GitLab на наличие новых задач и если он их обнаружит, то начнёт выполнять. Давайте подробнее разберём, как это происходит.

## Архитектура GitLab Runner

Два главных игрока:

- **GitLab**: хранит ваш код и файл `.gitlab-ci.yml`, управляет очередью задач и отправляет инструкции через API.
- **GitLab Runner**: установленное приложение, которое забирает задачи и выполняет их, используя выбранный исполнитель (например, `Docker` или `Shell`).

Runner подключается к GitLab через HTTPS, используя токен для безопасности. Он периодически спрашивает: «Есть ли для меня работка?» Если задача есть, он скачивает код, читает `.gitlab-ci.yml` и выполняет команды.

## Как это работает пошагово

1. **Регистрация**: вы регистрируете раннер в GitLab с уникальным токеном, указывая теги и тип исполнителя.
2. **Опрос очереди**: раннер проверяет, есть ли задачи, подходящие под его теги.
3. **Получение задания**: раннер загружает `.gitlab-ci.yml` и клонирует репозиторий.
4. **Выполнение**: раннер запускает задачу в выбранной среде (например, в Docker-контейнере).
5. **Отчёт**: раннер отправляет логи и артефакты обратно в GitLab.

Что такое `job_token`? Это временный токен, который GitLab создаёт для каждой задачи. Он обеспечивает безопасное общение между раннером, исполнителем и самим GitLab.

Понимание работы GitLab Runner помогает быстро находить и исправлять ошибки в пайплайнах. Например, если задача не запускается, вы знаете, где искать: в регистрации, тегах, исполнителе или конфигурации.

# Исполнители в GitLab Runner

Исполнители (executors) в GitLab Runner определяют среду и способ выполнения задач CI/CD. От запуска простых скриптов на сервере до деплоя в масштабируемых Kubernetes-кластерах — выбор исполнителя зависит от ваших задач, инфраструктуры и требований к безопасности. В этом шаге разберём четыре основных типа исполнителей, их технические особенности и сценарии использования.

> Это поможет вам в будущем выбрать подходящий для вашего проекта.

## Основные исполнители

### `Shell`

Shell-исполнитель выполняет команды непосредственно в операционной системе сервера, где установлен GitLab Runner. Runner использует системную оболочку (например, `sh` на Linux или `cmd`/`PowerShell` на Windows) для запуска скриптов, указанных в `.gitlab-ci.yml`.

**Как работает?**  
Команды выполняются в той же среде, где работает Runner, с доступом к файловой системе хоста, переменным окружения и установленным утилитам. Это означает, что задачи используют ресурсы сервера напрямую, без дополнительной изоляции.

**Когда использовать?**

- Для простых задач, таких как запуск локальных скриптов или выполнение команд, не требующих сложной настройки окружения.
- Подходит для проектов, где Runner установлен на выделенном сервере с заранее настроенным ПО.

**Потенциальные проблемы:**

- Управление зависимостями: если на сервере отсутствует нужная версия ПО (например, Python 3.8 вместо 3.9), пайплайн может завершиться ошибкой.
- Зависимость от хостовой среды: необходимо вручную устанавливать все зависимости.
- Безопасность: задачи, выполняемые с привилегиями Runner, могут случайно или злонамеренно повредить сервер.

### `Docker`

Docker-исполнитель запускает каждую задачу CI/CD в Docker-контейнере. Runner взаимодействует с Docker-демоном на хосте, чтобы создавать и управлять контейнерами на основе указанных в `.gitlab-ci.yml` образов.

**Как работает?**  
Для каждой задачи Runner создаёт новый контейнер из указанного Docker-образа (например, `python:3.9`). Контейнер изолирует задачу от хостовой системы, обеспечивая чистую среду с предустановленными зависимостями. После завершения задачи контейнер удаляется.

**Когда использовать?**

- Для большинства современных CI/CD-пайплайнов, где важны изоляция, воспроизводимость и управление зависимостями.
- Подходит для тестирования, сборки и деплоя приложений в стандартизированных окружениях.

**Потенциальные проблемы:**

- Docker-in-Docker: для сборки Docker-образов требуется привилегированный режим (`privileged = true`), что снижает безопасность.
- Ограничения ресурсов: интенсивные задачи могут перегрузить хост, если не настроены лимиты CPU/памяти.

### `Kubernetes`

Kubernetes-исполнитель запускает задачи CI/CD в подах на Kubernetes-кластере. Runner взаимодействует с API Kubernetes, чтобы создавать поды на основе спецификаций, указанных в конфигурации.

**Как работает?**  
Каждая задача выполняется в отдельном поде, созданном из Docker-образа, указанного в `.gitlab-ci.yml`. Kubernetes управляет ресурсами, масштабированием и изоляцией подов. После завершения задачи под удаляется, а логи передаются обратно в GitLab.

**Когда использовать?**

- Для проектов, интегрированных с Kubernetes, где требуется масштабируемость и оркестрация.
- Подходит для сложных CI/CD-пайплайнов.

**Потенциальные проблемы:**

- Ограничения ресурсов: без настройки `requests` и `limits` в подах задачи могут перегружать кластер.
- Сетевые политики: доступ к внешним сервисам может быть ограничен политиками Kubernetes.

### `SSH`

SSH-исполнитель выполняет команды на удалённом сервере через SSH-соединение. Runner подключается к указанному хосту, используя предоставленные учетные данные, и запускает скрипты в его среде.

**Как работает?**  
Runner устанавливает SSH-соединение с целевым сервером и выполняет команды, указанные в `.gitlab-ci.yml`, как если бы они запускались локально на этом сервере. Это позволяет работать с инфраструктурой, недоступной для других исполнителей.

**Когда использовать?**

- Для устаревших систем, где нельзя установить `Docker` или `Kubernetes`.
- Для специфических задач, требующих доступа к определённым серверам (например, деплой на legacy сервер без CI/CD-инструментов).

**Потенциальные проблемы:**

- Безопасность: хранение учетных данных в `config.toml` может быть рискованным.
- Ошибки подключения: проблемы с сетью или SSH-сервером могут прервать пайплайн.
- Ограниченная поддержка: SSH-исполнитель считается устаревшим и не рекомендуется для новых проектов.

## Как выбрать исполнителя?

Выбор исполнителя зависит от ваших задач, инфраструктуры и специфики проекта. Вот расширенная таблица с рекомендациями:

|**Цель**|**Рекомендация**|**Комментарий**|
|---|---|---|
|Простота настройки|`Shell`|Минимальная конфигурация, но требует осторожности из-за отсутствия изоляции.|
|Безопасность и изоляция|`Docker` или `Kubernetes`|Docker проще для локальных серверов, Kubernetes — для кластеров.|
|Масштабирование|`Kubernetes`|Идеально для больших проектов с распределённой нагрузкой.|
|Legacy-системы|`SSH`|Используйте только при отсутствии других вариантов, избегайте для новых проектов.|
|Минимизация затрат|`Shell` или `Docker`|`Shell` не требует доп. ресурсов, `Docker` экономичен на одиночных серверах.|
|Воспроизводимость окружения|`Docker` или `Kubernetes`|Обеспечивают одинаковую среду для выполнения задач.|

> **Совет:** для большинства проектов начните с Docker — он сочетает простоту настройки, изоляцию и гибкость.

# Настройка GitLab Runner: `config.toml`

`config.toml` — это конфигурационный файл GitLab Runner. Он определяет, как Runner взаимодействует с GitLab, какие исполнители использует, сколько задач может выполнять одновременно и много чего ещё. Мы разберём основные настройки и приведём примеры, чтобы вы могли адаптировать Runner под свои проекты.

> **Примечание**: обычно вы не будете часто редактировать конфигурационный файл Runner’a вручную или писать с нуля. Регистрация Runner’a в GitLab обычно происходит через команду `gitlab-runner register`, что гораздо удобнее. Однако, чтобы понимать структуру и основные параметры конфигурации, мы подготовили этот шаг. Рано или поздно вам всё равно придётся что-то изменить или настроить.

## Структура `config.toml`

Конфигурационный файл написан в формате `TOML`. Он состоит из двух основных частей:

- **Глобальные настройки**: задают общее поведение всех Runner’ов на сервере.
- **Секции `[[runners]]`**: описывают параметры каждого зарегистрированного Runner’а, включая его подключение к GitLab и тип исполнителя.

Пример минимальной конфигурации**:**

```ini
concurrent = 1
check_interval = 10

[[runners]]
  name = "gitlab-runner"
  url = "https://cicdcourse.gitlab.yandexcloud.net/"
  token = "glrt-xxxxxxxxxxxxxxxx"
  executor = "docker"
  [runners.docker]
    image = "ubuntu"
    privileged = false
    volumes = ["/cache"]
```

Она определяет:

- Максимум 1 одновременно выполняющуюся задачy (`concurrent = 1`).
- Проверку новых задач каждые 10 секунд (`check_interval = 10`).
- Один Runner с именем `gitlab-runner`, использующий Docker-исполнитель с базовым образом `ubuntu`.

## Глобальные настройки

### `concurrent`

Определяет максимальное количество задач, которые Runner может выполнять одновременно на сервере**:**

```ini
concurrent = 4
```

Это позволяет контролировать нагрузку на сервер, предотвращая перегрузку при большом количестве задач.

### `**log_level**`

Задаёт yровень детализации логов Runner’а. Возможные значения: `debug`, `info`, `warn`, `error`, `fatal`, `panic`**:**

```ini
log_level = "info"
```

## Секция `[[runners]]`

### `name`

Имя Runner’а, отображаемое в интерфейсе GitLab. Оно помогает идентифицировать его в списке доступных в настройках проекта**:**

```ini
name = "gitlab-runner"
```

### `url`

URL-адрес экземпляра GitLab, к которому подключается Runner. Он использует этот URL для взаимодействия с API GitLab, чтобы получать задачи и отправлять результаты**:**

```ini
url = "https://cicdcourse.gitlab.yandexcloud.net"
```

### `token`

Уникальный токен, связывающий Runner с GitLab. Он генерируется при регистрации Runner’а в веб-интерфейсе. Он использует его для аутентификации при получении задач**:**

```ini
token = "glrt-xxxxxxxxxxxxxxxx"
```

### `executor`

Тип исполнителя, который Runner использует для выполнения задач. Определяет, в какой среде будут выполняться команды из `.gitlab-ci.yml`. Каждый исполнитель имеет свои поднастройки (например, `[runners.docker]` или `[runners.ssh]`). Пример**:**

```ini
executor = "docker"
```

### `[runners.docker]`

Поднастройки для Docker-исполнителя, определяющие поведение контейнеров. Runner использует эти параметры для создания и управления Docker-контейнерами.

Основные параметры**:**

- `image`: базовый Docker-образ для контейнеров.
- `privileged`: включает привилегированный режим для контейнеров.
- `volumes`: монтирует директории или тома в контейнер.
- `memory`: ограничивает использование памяти.
- `cpus`: ограничивает использование CPU.

Пример**:**

```ini
[runners.docker]
  image = "python:3.9"
  privileged = false
  volumes = ["/cache"]
  memory = "512m"
  cpus = "1.0"
```

## Как редактировать?

Файл `config.toml` обычно находится по пути `/etc/gitlab-runner/config.toml` (на Linux). После внесения изменений перезапустите Runner, чтобы применить настройки:

```bash
sudo gitlab-runner restart
```

Или, если используется systemd:

```bash
sudo systemctl restart gitlab-runner.service
```
